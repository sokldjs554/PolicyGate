# 아키텍처 및 설계 결정 기록 (ADR)

이 문서는 PolicyGate의 구조와, 각 결정을 내린 이유·검토했던 대안·감수한
트레이드오프를 기록합니다.

## 전체 구조

```
                ┌──────────────────────────────────────────────────────┐

                │  rbac.py — 선언적 권한 매트릭스 (403/409/400/422 매핑)  │
                └──────────────────┬───────────────────────────────────┘
                                   │ 벤더 중립 모델(FirewallRule)만 통과
                ┌──────────────────▼───────────────────────────────────┐
                │  workflow/service.py — 라이프사이클 오케스트레이션        │
                │  workflow/state.py   — 상태 머신(전이 화이트리스트)       │
                └───────┬──────────────────────────┬────────────────────┘
                        │                          │
        ┌───────────────▼────────────┐  ┌──────────▼───────────────────┐
        │ analysis/                  │  │ adapters/                    │

        │  simulator.py 시뮬레이션    │  │  verify.py netns 실배포 검증   │
        │  logscan.py   로그 대조     │  │  base.py 인터페이스+실행 이음새 │

                │ rule.py — 도메인 모델 + 정수 구간 관계 연산              │
```

계층 간 의존은 위에서 아래로만 흐릅니다. 분석 엔진은 Flask도 SQLite도 모르고,
어댑터는 워크플로를 모릅니다. 각 조각을 독립적으로 테스트·교체할 수 있는 이유입니다.

## ADR-1. 분석은 벤더 중립 모델 위에서 수행한다

**결정.** 정책을 iptables 문법이 아니라 5-tuple 추상 모델(`FirewallRule`)로
정규화한 뒤, 모든 분석·워크플로가 이 모델 위에서만 동작하게 했다.

**이유.** 방화벽 벤더가 무엇이든(iptables, nftables, 상용 장비) 정책의 의미론은
"출발지·목적지·프로토콜·포트·액션의 first-match"로 동일하다. 분석을 벤더 문법에
결합하면 벤더가 늘 때마다 분석기를 다시 짜야 한다.

**트레이드오프.** 벤더 고유 기능(존, NAT, 애플리케이션 시그니처)은 공통 모델로
표현되지 않는다. 이는 모델 확장(필드 추가 + 관계 연산 확장)으로 대응한다.

## ADR-2. Shadowing 판정은 보수적으로 — 정확성 > 검출률

**결정.** `SHADOWED`/`REDUNDANT` 확정 판정은 **단일 선행 정책에 의한 완전
포함**일 때만 내린다. 여러 선행 정책의 합집합이 뒤 정책을 가리는 경우(union
shadowing)는 확정 보고하지 않고, 부분 겹침을 `CORRELATED`(주의)로 보고한다.

**검토한 대안.** 정책 셋 전체를 구간 트리/BDD(Binary Decision Diagram)로
모델링하면 union shadowing까지 완전 판정이 가능하다. 그러나 구현 복잡도가 크게
뛰고, 판정 로직 자체의 버그 위험이 커진다.

**이유.** 감사 도구의 신뢰는 거짓 양성(false positive)이 없을 때 생긴다.
"도구가 SHADOWED라고 하면 정말 죽은 정책이다"를 보장하는 쪽이, 검출률 몇 %를
더 얻는 것보다 운영 가치가 크다고 판단했다. 미검출 영역은 CORRELATED 경고로
사람의 눈을 유도한다.

## ADR-3. 포트 연산은 정규형 불변식 위에 세운다

**결정.** 모든 포트 명세는 입력 시점에 정렬·병합된 정규형(겹치지 않는 구간
목록)으로 변환하고, 포함/중첩 판정은 그 불변식을 전제로 한 O(n+m) 투 포인터
스캔으로 구현했다.

**이유.** "80-90,85-100"과 "80-100"이 같은 정책인데 다르게 판정되면 분석
전체가 무너진다. 정규화를 입구(팩토리) 한 곳에 강제하면 이후의 모든 연산이
단순해지고, 경계 버그(±1)를 테스트로 고정하기 쉽다.

## ADR-4. 절차는 상태 머신으로 강제한다

**결정.** 신청의 상태 전이를 화이트리스트(`ALLOWED_TRANSITIONS`)로 선언하고,
모든 전이가 `assert_transition`을 통과하게 했다. 추가로 신청자 본인 승인을
금지했다(직무 분리).

**이유.** "승인 없이 배포하지 마세요"라는 규칙은 문서에 있으면 언젠가 깨진다.
코드에 있으면 깨질 수 없다. 보안 통제는 사람의 선의가 아니라 구조로 보장해야
한다는 원칙을 그대로 구현한 것이다.

**결과.** `deploy()`는 `approved` 상태가 아니면 예외를 던지고, 반려된 신청은
어떤 경로로도 되살릴 수 없다(종결 상태). 테스트 `test_deploy_without_approval_fails`
등이 이 보장을 회귀 방지한다.

## ADR-5. 배포는 선언형 전체 교체(declarative full-set)

**결정.** 어댑터는 개별 룰을 추가/삭제하는 명령을 만들지 않고, 항상 **원하는
최종 상태 전체**를 생성해 원자적으로 교체한다(iptables-restore, `nft -f` +
flush).

**검토한 대안.** `iptables -A/-D`로 증분 반영하면 반영 시간은 짧다. 그러나
관리 시스템 밖에서 누군가 장비를 만지는 순간 시스템의 상태와 장비의 상태가
어긋나고(드리프트), 그 어긋남은 조용히 누적된다.

**이유.** 시스템이 곧 단일 진실 공급원(SSOT)이어야 한다. 전체 교체는 드리프트를
매 배포마다 자동으로 소거하며, iptables-restore는 원자 적용이라 반쪽 상태도
없다. 모든 룰에 `rule_id` 주석을 심어 장비 → 시스템 역추적도 가능하게 했다.

## ADR-6. 시스템 명령 실행을 이음새(seam)로 분리

**결정.** 어댑터가 subprocess를 직접 호출하지 않고 `CommandRunner` 객체를
주입받는다. 러너가 없으면 설정 생성만 하고 장비에 손대지 않는다(안전 기본값).

**이유.** 실장비 없이 배포 로직 전체를 테스트하기 위해서다. 또한 "실수로 실행"의
기본값을 차단하는 효과가 있다 — 명시적으로 러너를 주입해야만 실제 반영이 일어난다.

## ADR-7. 저장소는 SQLite, 그러나 교체 가능한 계층으로

**결정.** 저장을 `Store` 클래스 하나에 격리하고 SQLite(표준 라이브러리)로
구현했다.

**이유.** 이 프로젝트 규모에서 외부 DB는 운영 부담만 늘린다. 다만 워크플로
코드가 SQL을 직접 만지지 않게 격리했으므로, PostgreSQL 전환 시 `Store`만
재구현하면 된다. 감사 로그는 UPDATE 경로 없이 INSERT만 존재하는 append-only
테이블로 설계했다.

## ADR-8. 프레임워크 선택 — Flask + click + 표준 라이브러리

**결정.** 분석 엔진은 표준 라이브러리만 사용(ipaddress, dataclasses), API는
Flask, CLI는 click.

**이유.** 핵심 가치(정확한 판정)는 의존성이 아니라 알고리즘에 있다. 의존성이
가벼울수록 보안 도구 자체의 공급망 표면적이 줄고, 어떤 환경에서든 돌릴 수 있다.

## ADR-9. RBAC는 API 경계에서, 업무 불변식은 도메인에서

**결정.** 인가(누가 무엇을 할 수 있는가)는 선언적 권한 매트릭스
(`rbac.PERMISSIONS`) 하나에 모으고 API 경계(`api.py`)에서만 검사한다.
도메인 서비스(`PolicyService`)는 직무 분리(신청자 ≠ 승인자) 같은
**업무 불변식**만 담당한다. 역할은 클라이언트가 헤더로 주장하는 값이 아니라
서버측 users 테이블에서 조회한다.

**검토한 대안.** (a) 서비스 메서드마다 권한 검사 — 인가 정책이 코드 전역에
흩어지고, CLI·배치 등 API를 거치지 않는 호출자와 충돌한다. (b) 데코레이터로
엔드포인트에 역할 하드코딩 — 역할과 권한이 1:1로 굳어 새 역할 추가 시 전체
수정이 필요하다.

**이유.** "권한 정책이 어디 있는가?"에 대한 답이 파일 하나(매트릭스)여야
감사자도 개발자도 정책을 검토할 수 있다. 역할(누구)과 권한(무엇)을 분리하면
새 역할 추가가 매트릭스 수정만으로 끝난다. 인가 실패(403)와 절차 위반(409)을
예외 타입으로 구분해, 감사 관점에서 "권한 없는 시도"와 "절차를 어긴 시도"가
분리 관측된다.

**부트스트랩 문제.** 사용자 등록이 admin 전용인데 최초에는 admin이 없다.
users 테이블이 비어있을 때 1회에 한해 부트스트랩 admin을 생성하고 그 사실을
감사에 남기는 방식으로 해결했다 (교착 방지 + 추적 가능).

## ADR-10. 배포 검증은 격리 namespace의 실제 커널에서

**결정.** 배포 전 검증을 "문법 검사"가 아니라 **실제 적용**으로 한다.
`unshare --net`으로 격리된 network namespace를 만들어 그 안의 커널에
iptables-restore를 수행하고, iptables-save readback으로 모든 rule_id와
기본 정책(DROP)을 대조한다. 검증 실패 시 배포는 중단되고 상태는 approved로
유지된다(재시도 가능).

**검토한 대안.** (a) `iptables-restore --test` — 문법만 검사하고 커널 모듈
가용성/버전 차이를 잡지 못한다. (b) `ip netns add` — iproute2 의존 +
namespace 정리 실패라는 오류 부류가 생긴다. (c) Docker 컨테이너 — 데몬
의존성이 크고 CI에서 DinD 문제가 있다.

**이유.** "설정을 생성했다"와 "커널이 받아줬다" 사이에는 실제 간극이 있다
(지원되지 않는 매치, 모듈 부재). unshare는 util-linux만으로 동작하고
프로세스 종료와 함께 namespace가 소멸하므로 **정리 실패라는 상태 자체가
없다**. 호스트 방화벽에는 어떤 영향도 없다. 검증 실패를 예외가 아닌 결과
객체로 반환해 워크플로가 '상태'로 다루고 감사에 남긴다.

**한계.** 격리 namespace의 커널은 호스트와 같은 커널이므로, 대상 장비의
커널이 다르면 검증이 완전하지 않다. 실환경 확장 시 장비별 readback 검증으로
보완한다(README 확장 방안 참조).

## ADR-11. 자료구조와 복잡도 — 정수 구간, 그리고 Trie/Interval Tree의 자리

**결정.** 관계 연산의 핵심 자료구조로 (a) CIDR의 정수 구간 표현
`(network_address, broadcast_address)`, (b) 정렬·병합된 포트 구간 목록
(정규형), (c) 불변 Rule 객체 + 생성 시 1회 계산되는 구간 캐시를 선택했다.
관계 분석 루프에는 정수 비교 4회짜리 빠른 탈락 경로를 두었다.

**이유.** CIDR은 정의상 연속된 주소 구간이므로 포함(⊇)과 교차(∩≠∅)가
정수 대소 비교로 정확히 환원된다. ipaddress 객체 연산(쌍당 수 µs) 대비
수십 배 빠르며 의미 손실이 없다. 포트는 정규형 불변식("항상 정렬·병합됨")을
입구에서 강제하면 이후 모든 연산이 O(n+m) 투 포인터로 단순해지고, 같은
의미의 다른 표기("80-90,85-100" vs "80-100")가 같은 판정을 받는다.

**복잡도 정리.**

| 연산 | 복잡도 | 비고 |
|---|---|---|
| 정규화/구간 변환 | O(1)/정책 | 생성 시 1회 |
| 쌍 관계 판정 | O(p), p=포트 구간 수 | 보통 p≤2 |
| 신규 신청 검증 | O(n) | 10k 셋에서 <10ms |
| 전체 관계 분석 | O(n²) + 빠른 탈락 | 10k → 3.4s 실측 |
| 로그 대조 | O(m·n) 최악 | first-match 조기 종료 |

**Trie / Interval Tree 개선 경로.** O(n²)의 본질은 후보 쌍 전수 조사다.
- Interval Tree(dst 구간): 정책당 겹침 후보를 O(log n + k)로 조회 →
  전체 O(n log n + K), K=실제 겹침 쌍 수. 현실 정책 셋에서 K ≪ n².
- Patricia Trie(src/dst CIDR): CIDR 포함 관계 = 트라이 조상-자손 관계.
  신규 신청 검증이 O(32) 상수 시간으로 — 라우팅 테이블 LPM과 동일 원리.
- 적용 지점은 `_detect_relations`와 `validate_new_rule` 둘로 격리되어 있어
  인덱스 도입이 국소적이다.

**도입하지 않은 이유(이것도 결정이다).** 실측 10,000건 3.4초는 대부분
조직의 정책 규모를 여유 있게 커버한다. 인덱스는 유지 비용(삽입/삭제 시
갱신)과 코드 복잡도를 수반하므로, 필요해지기 전에는 단순한 코드가 옳다.
벤치마크가 그 "필요해지는 시점"을 수치로 감시한다.

## ADR-12. ID 채번 — COUNT(*)에서 원자 카운터로 (동시성 버그의 기록)

**결정.** 요청/정책 ID 채번을 `SELECT COUNT(*)` 기반에서 카운터 테이블의
`UPDATE ... RETURNING` 원자 증가로 교체했다.

**배경.** 최초 구현은 행 수+1을 ID로 썼다. 동시 신청 테스트(스레드 20개
동시 출발)를 작성하자 두 요청이 같은 ID를 받는 race가 재현됐다 — 채번과
삽입이 별개 연산이라 그 사이에 다른 스레드가 끼어들 수 있었다.

**이유와 교훈.** 동시성 버그는 코드 리뷰보다 "동시성을 강제하는 테스트"가
잘 찾는다. 수정 후 같은 테스트가 회귀 방지선이 됐다
(`tests/test_scale_concurrency.py`). UUID 대안도 검토했지만, 운영자가
읽고 부르는 ID(FW-0042)의 가독성 가치가 커서 순번을 유지했다.

## ADR-13. 감사 로그 — before/after 스냅샷과 append-only

**결정.** 모든 변경 이벤트에 timestamp/actor/action에 더해 변경 전(before)/
후(after) 상태 스냅샷(JSON)을 남긴다. audit_log 테이블에는 UPDATE/DELETE
경로가 없다(append-only).

**이유.** "누가 언제 무엇을 했다"만으로는 사고 조사에서 부족하다 —
"그래서 상태가 어떻게 바뀌었는가"가 복원되어야 한다. before/after가 있으면
감사 로그만으로 임의 시점의 정책 상태를 재구성할 수 있고, 롤백의 정당성
(무엇으로 되돌렸는지)도 증명된다. append-only는 부인 방지의 최소 조건이며,
실서비스에서는 외부 WORM 스토리지 스트리밍으로 보강한다.
