# 배포 검증 (Deployment Verification)

설정 파일을 만드는 것과 커널이 그 설정을 받아주는 것은 다른 문제입니다.
문법은 맞지만 커널 모듈이 없어 거부되거나, 버전에 따라 지원되지 않는 매치가
섞이는 경우가 있습니다. PolicyGate는 배포 전에 이 간극을 실제 적용으로
확인합니다.

## 동작 방식

1. `unshare --net`으로 격리된 network namespace를 만듭니다. 호스트 방화벽에는
   어떤 영향도 없고, 프로세스가 끝나면 namespace도 함께 사라지므로 정리 실패
   같은 상태가 아예 없습니다.
2. 그 안에서 `iptables-restore`로 생성된 설정을 커널에 적용합니다.
3. `iptables-save`로 적용 결과를 다시 읽어(readback) 다음을 대조합니다.
   - 모든 rule_id 주석이 존재하는가
   - 기본 정책이 DROP(deny-by-default)인가
4. 하나라도 어긋나면 검증 실패로 배포가 중단됩니다. 신청 상태는 approved로
   유지되므로 원인을 고친 뒤 재시도할 수 있습니다.

restore와 save는 반드시 하나의 프로세스(같은 namespace) 안에서 연속
실행해야 합니다. 따로 띄우면 서로 다른 namespace가 만들어집니다.
구현은 `policygate/adapters/verify.py`, 설계 배경과 대안 비교(--test 플래그,
ip netns, Docker를 쓰지 않은 이유)는 [ARCHITECTURE.md](ARCHITECTURE.md)의
ADR-10에 있습니다.

## 실행 결과

### 정상 케이스

```console
$ sudo python -m policygate.cli verify sample_data/policies.csv
검증 통과: 모든 정책이 커널에 적용됨 (기본 정책 DROP 확인)
커널 적용 확인: 정책 12건 전부 readback 대조 통과
```

readback이 진짜 커널 왕복이라는 증거: 입력에 넣은 `ESTABLISHED,RELATED`가
커널 정규화를 거쳐 `RELATED,ESTABLISHED`로 재정렬되어 돌아오고, 출력 헤더도
우리가 만든 파일이 아닌 `# Generated by iptables-save v1.8.10`입니다.

```console
(readback 일부)
# Generated by iptables-save v1.8.10 (nf_tables)
*filter
:FORWARD DROP [0:0]
-A FORWARD -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
-A FORWARD -s 10.0.0.0/16 -d 10.20.0.10/32 -p tcp -m multiport --dports 443 -m comment --comment FW-0001 -j ACCEPT
```

### 실패 케이스 1 — 정책 누락 탐지

설정에서 정책 하나를 빼고 검증하면 정확히 그 정책이 잡힙니다.

```console
(설정에서 FW-0012를 뺀 뒤 검증)
ok = False | missing = ['FW-0012']
검증 실패: 누락된 정책: FW-0012
```

### 실패 케이스 2 — 커널이 거부하는 설정

```console
(존재하지 않는 체인을 참조하는 깨진 설정)
ok = False | applied = False
iptables-restore v1.8.10 (nf_tables): Chain 'NOWHERE' does not exist
```

## 워크플로 연동

API에서 배포 시 `{"verify": true}`를 주면 검증을 통과해야만 배포됩니다.

```bash
curl -X POST localhost:8080/api/requests/REQ-0001/deploy -H 'X-User: bob' \
  -H 'Content-Type: application/json' -d '{"adapter":"iptables","verify":true}'
```

- 검증 통과: 응답의 `verification.ok = true`, 감사에 `DEPLOY_VERIFIED` 기록
- 검증 실패: 409 반환, 상태는 approved 유지, 감사에 `DEPLOY_VERIFY_FAILED` 기록

## 테스트와 CI

- `tests/test_verify.py` — 커널 적용/readback 대조, 누락 탐지, 깨진 설정,
  워크플로 연동까지 5개 테스트. 환경이 안 되면(root 아님 등) 자동 skip.
- CI의 `netns-verify` job은 이 테스트가 skip 없이 실제 수행됐는지를 별도로
  확인합니다.

## 한계

격리 namespace의 커널은 호스트와 같은 커널입니다. 배포 대상 장비의 커널이나
모듈 구성이 다르면 검증이 완전하지 않으며, 상용 방화벽 장비에서는 검증하지
않았습니다. 다수 장비 환경에서는 장비별 readback 대조로 보완하는 것이 확장
방향입니다.
